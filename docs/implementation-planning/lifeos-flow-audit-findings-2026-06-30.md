Status: Diagnostic audit plus first repair batch status

## Severity key

- P0: A normal user can get stuck, lose the next step, or see a false system truth.
- P1: Required flow exists only partially, is misleading, or drops important state from the cockpit.
- P2: Useful edge case, parity, persistence, or proof gap.

## Root cause synthesis

1. The pass implemented the visual handoff before completing the state machine. The screens look like the new product, but several buttons are not backed by lifecycle transitions that land in the next intended place.
2. The cockpit view model is too narrow. It filters by active area and `scheduled` block status in places where the flow map needs global scope, running work, missed work, and blocked work.
3. Session execution has split truth. React owns timer state, the reducer owns block/session state, and persistence owns session outcomes. They are not connected enough to represent running, paused, actual duration, stuck, and missed recovery.
4. Persistence is best-effort but invisible. That is acceptable for local-first recovery only if the UI says when account sync or browser storage is not trustworthy.
5. Health and Google Calendar are truth-bearing surfaces, but the cockpit currently treats them as static explanatory surfaces. That creates false confidence for Health and dead-end capability for Google approval.
6. The current E2E suite protects hierarchy and render parity, not full journeys. It passes even when Execute cannot complete a session, Review cannot close a day, and All areas is only partially global.
7. Display fallbacks leak into data. The fabricated `area-default` is useful for rendering but unsafe as a real workflow area id.
8. Several older behaviors were partially retained instead of deliberately resolved. `Organize after save` is the clearest example: the control exists, but the branch does not.

## What still works

- Workflow routes render through the one-screen cockpit router.
- Capture -> Triage works for normal text input in the local/mock path.
- Triage Do today and Someday create active/backlog tasks.
- Local hour-rail planning creates a visible scheduled block.
- Theme, active area switching, count-bearing nav, header add/recolor, and the handoff visual hierarchy are present.
- The current checked-in E2E suite passes as a visual/router smoke suite, but it is not sufficient flow proof.

## Recommended repair order

1. Keep Google Calendar approval bridging deferred until env-backed provider work is in scope again (F17).
2. Keep AI/env-dependent capture failure/manual-routing, audio transcription, and weekly calibration deferred until parser/provider work is in scope again (F13/F30/F31 where relevant).
3. Keep durable draft-decision audit, retry queues, and broader reload-proof fixture coverage as product follow-up decisions; the scoped account persistence for manual proposals, unplan, review task transitions, session start, and daily review save is implemented.

## Test strategy changes needed

- Keep the current handoff E2E specs as visual smoke tests.
- Add journey specs that assert the next intended place, not just visible hierarchy.
- Add reducer/view-model tests for global overview lists, running blocks, missed blocks, stuck sessions, and review recovery buckets.
- Add storage/account degradation tests that assert visible local-only/sync-failed copy near the affected action and retry behavior if a retry queue is introduced.
- Add a Google approval bridge test that proves local planning never writes externally until the final explicit confirmation.
