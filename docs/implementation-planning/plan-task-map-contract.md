# Task Node Map / Progression — v1/v2 Contract

Status: v1 shipped; the remaining v2 product/data architecture proposal awaits explicit IMPLEMENT/MERGE/REJECT/DEFER disposition. Read-only planning artifact; current code, REQUIREMENTS, and ADRs override stale v1 proposal details below.
Author role: Product/Data Architect. Scope: FR-031 (v1), v2 sketch, sequencing, risks.
Strategic sources: owner-ratified staging note in issue #292 comment (2026-07-05), ADR 0002's trust ladder/invariants as amended by ADR 0005, ADR 0003, and the parse breakdown contract (`packages/schemas/src/parse-capture.ts`). Current REQUIREMENTS and ADRs remain implementation authority.

## 0. Framing decisions (load-bearing — read first)

1. **v1 is an annotation + persistence layer over the EXISTING `ParseCaptureBreakdownSchema`, not a parallel node model.** The breakdown already delivers 2–7 ordered steps (matches the ~7-node cap), `depends_on_orders`, `on_critical_path`, and `kickstart_step`. The map REUSES those. v1 adds only three per-node fields (`is_speculative`, `is_optional`, `refrain_note`), the one-pass approve gate, and persistence. The v0 progression rail (render of the raw breakdown) is therefore literally the v1 failure-degradation target: **map = breakdown superset**.
2. **One-pass approve is L1, not L2.** Approving the whole map once = a single L1 approval of one suggestion _instance_; each AI-proposed revision is a fresh L1 approval (propose → validate → approve → persist). This is spelled out in FR-031 so no reviewer reads "one-tap approve" as an L2 (pre-filled default + auto-execute) violation of D1's "no slice ships at L2+ on day one."
3. **The v2 gate metric dictates v1 instrumentation granularity.** The proposed gate "breakdown-node acceptance >= 70%" is _per-node_, so v1 MUST record node-level resolution, not just a map-level approve. Gate formula frozen here (§4.4) and wired to the §3.4 writes.
4. **Sequencing: v1 waits for S3 (#255) and lands additively on top; it does NOT ride S3's schema bump.** Same parse surface; #255 warns parallel work "guarantees conflict." Chain: **S2 → S3 (#255) → v1**. v1 is its own FR + DATA_MODEL amendment (§5).
5. **Data model: a jsonb field on `tasks` for v1; a normalized `task_edges` table for v2.** Caps (7 nodes, depth 1) make a denormalized `progression_map jsonb` correct and additive (NS-INV-2). Breakdown is currently ephemeral parse output with no `tasks` column, so persisting the approved map at triage-accept is a genuine new additive column.

## 1. FR-031 — AI-Drafted Task Node Map (v1)

_(exact REQUIREMENTS.md FR format)_

### FR-031 — AI-Drafted Task Node Map

**Priority:** MUST

**Stage:** Stage 1 (epic #251), post-S3 additive slice (new FR-first per ADR 0002 D5)

When a capture parses into a task with a multi-step breakdown, the system presents that breakdown as an AI-drafted, one-pass-approvable **progression map** — a lay-of-the-land node view that satisfies the operator's map-before-motion prerequisite in seconds, so planning is outsourced rather than suppressed. The map is a rendering + light annotation of the existing `ParseCaptureBreakdown`, not a new decomposition.

**MUST**

- The map renders the existing breakdown steps as ordered nodes; node identity, order, dependencies, and critical-path membership derive **only** from the already-produced `breakdown.steps` (`order`, `depends_on_orders`, `on_critical_path`). No second full LLM decomposition is performed to build the map (NFR-001).
- Enforce hard caps **regardless of trust rung**: at most **7 nodes**, dependency **depth 1** (flat ordered list with pairwise `depends_on`, no nested subtrees), and a **collapsed-to-next-node default** view (the next actionable node is expanded; the rest collapse).
- **On-demand, never background.** Map _annotations_ are generated by piggybacking the already-triggered capture-parse call (no second decomposition — §3.1); map _materialization/rendering_ happens only when the user opens the map or accepts the task at triage; the _regeneration_ call is user-triggered only. No step of the pipeline runs as a background job (NFR-001, NFR-005).
- The map is approved in **one pass** ("right enough to start?"). On approve, the map persists on the task and node 1 becomes the active/lit node. This is an **L1 approval** of a single suggestion instance (ADR 0002 D1): the whole map is one proposal approved once; it is NOT an L2 pre-filled auto-execute default. No node executes without this approval (NS-INV-4).
- Map revisions are **AI-proposed diffs from execution evidence**, each presented as a fresh L1 proposal the user approves or rejects (propose → validate against strict schema → approve → persist). Revisions never mutate the persisted map silently (NS-INV-4). _(In v1 the only revision trigger is explicit user-requested regeneration; evidence-driven auto-proposal is a v2 non-goal — see §4.)_
- **Speculative nodes are visually marked.** Any node the model is not confident belongs in the committed path carries `is_speculative = true` and renders with a distinct "guess" affordance so the user can see plan vs. guess (consistent with FR-004 facts-vs-assumptions doctrine).
- **Refrain annotations** (per-node "what not to do yet" / operator-profile cautions) are populated **only** through the single context-assembly choke point `apps/web/src/lib/ai/contextAssembly.ts` (NS-INV-1). The map slice wires no charter/profile plumbing of its own; it consumes the assembled context via that module's defined extension point.
- The surface is **born instrumented** (NS-INV-3): it writes `suggestion_records` / `override_records` from first merge using #235 vocabulary — a stable policy identifier `task_map.v1`, a versioned zod schema in `packages/schemas`, node-level resolution recorded (see §3.4). Writes are fire-and-forget; a learning-write failure must not break the approve flow.
- On any AI failure (parse error, schema-invalid map annotation, timeout), the surface **degrades to the plain breakdown rail** (the v0 render of raw `breakdown.steps`) with raw capture preserved (NFR-004). A missing/failed map is never a dead end (ADR 0003 UX-INV-3).

**SHOULD**

- The collapsed default and the lit-node progression should read as calm coaching, not gamified pressure (ADR 0003 D3 taste contract).
- Optional (non-critical-path) nodes carry `is_optional = true` and de-emphasize visually so the critical path is legible at a glance.
- The map should record `map_approved_at` and node execution-start timestamps to make overplanning-dwell measurable (§6).

**NON-GOALS**

- No cross-task edges, no DAG spanning multiple tasks, no dependency depth > 1 (that is v2 `task_edges`).
- No multi-task extraction from one capture (v2).
- No autonomous / evidence-triggered map revision (v2 map-maintenance loop).
- No new decomposition model call to build the map; no scheduling, no calendar writes from the map.
- No AI-authored map that persists without user approval (NS-INV-4).
- No gamification scoring, XP, streaks, or reward mechanics.

## 2. v1 Data Model (DATA_MODEL.md format, NS-INV-2 additive-only)

### 2.1 What fits in existing contracts (extend, don't add)

- **Parse-result / breakdown JSON contract** — extend the existing `ParseCaptureBreakdownStepSchema` in `packages/schemas/src/parse-capture.ts` with three optional per-step fields. This is a **versioned additive extension**, the same mechanism S3 (#255) uses for `person_mentions`; v1 takes the _next_ `schema_version` increment after S3 merges (§5). No existing field is altered or renamed (NS-INV-2).

```
ParseCaptureBreakdownStepSchema (additive fields)
  is_speculative   boolean   default false   -- model unsure this node belongs in the committed path
  is_optional      boolean   default false   -- node is off the critical path and skippable
  refrain_note     string    nullable        -- per-node "what not to do yet"; populated only via contextAssembly.ts (NS-INV-1)
```

Existing fields reused as-is (do NOT duplicate): `on_critical_path` IS the "critical" flag; `kickstart_step` / task `first_tiny_step` IS the two-minute move; the flat `steps[]` list with `depends_on_orders` IS the depth-1 node graph.

### 2.2 What needs an additive column

Breakdown is currently **ephemeral** parse output — there is no column persisting it on `tasks`. Persisting the _approved_ map at triage-accept is therefore a genuine new additive column on the existing `tasks` table (section 4.5), preferred over a new table per section 10/11 ("prefer fields"). Caps (7 nodes, depth 1) keep it small and denormalized.

### 4.5 `tasks` — additive columns (target shape, FR-031 / task-map v1)

Adds to the existing `tasks` table without altering or renaming any existing column (NS-INV-2). Export coverage is inherited automatically — `tasks` is already in `USER_DATA_EXPORT_TABLES` (INV-2), so no new export registration is needed; the guard test in `engineeringInvariants.test.ts` stays green.

| Column             | Type                 | Notes                                                                               |
| ------------------ | -------------------- | ----------------------------------------------------------------------------------- |
| progression_map    | jsonb nullable       | approved map; zod `TaskProgressionMap` (see §3.2). null = no map / degraded to rail |
| map_status         | text nullable        | check in (`draft`, `approved`, `superseded`); null when no map                      |
| map_schema_version | text nullable        | the parse `schema_version` the map was minted under (audit / migration)             |
| map_approved_at    | timestamptz nullable | set at one-pass approve; anchors overplanning-dwell measurement (§6)                |

- Node completion state lives **inside** `progression_map.nodes[].completed_at` (jsonb), not as separate columns — avoids state explosion (section 11) and stays within depth-1 caps.
- Standard owner RLS (section 8), unchanged. No new table ⇒ no new RLS policy, no new two-user isolation test beyond existing `tasks` coverage.
- **No `task_edges` table in v1** (that is v2, §4).

## 3. v1 AI Surface Spec

### 3.1 Prompt contract (versioned)

- The map does **not** get its own prompt/model call in the common path. The breakdown is already produced by `parse_capture.v3` (the post-S3 version — see §5); FR-031 adds map-annotation instructions to that same prompt so `is_speculative` / `is_optional` / `refrain_note` come back in the same structured response. Refrain content is injected via `contextAssembly.ts` (NS-INV-1), not hand-plumbed into the prompt file.
- Only the explicit user-requested **regeneration** path issues a distinct call, under prompt id `task_map_regen.v1`, reusing the parse response format. `store: false` (NFR-003), cost-optimized model tier (NFR-001).

### 3.2 Strict response zod schema (`packages/schemas`)

The annotated step schema (§2.1) is the wire contract. The persisted map is a derived, validated shape:

```
TaskProgressionNodeSchema = z.object({
  order:            z.number().int().min(1),
  title:            z.string().min(1),
  two_minute_move:  z.boolean(),                    // true for the node carrying kickstart_step / first_tiny_step
  on_critical_path: z.boolean(),                    // reused from breakdown
  is_optional:      z.boolean().default(false),
  is_speculative:   z.boolean().default(false),
  depends_on_orders:z.array(z.number().int().min(1)),// depth-1: refers only to sibling orders
  refrain_note:     z.string().nullable(),          // from contextAssembly.ts only
  completed_at:     z.string().datetime().nullable().default(null),
})

TaskProgressionMapSchema = z.object({
  schema_version: z.literal("task_map.v1"),
  nodes:          z.array(TaskProgressionNodeSchema).min(1).max(7),   // ~7-node cap enforced in schema
  sequence_summary: z.string().nullable(),
  generated_from:   z.enum(["parse", "regen"]),
})
```

Validation is as strict as existing parse contracts; invalid map output is rejected → degradation (§3.3). The `.max(7)` makes the node cap a schema invariant, not just UI.

### 3.3 Failure degradation

Ordered fallback, each visible per NFR-004 / UX-INV-3:

1. Map annotation call/regeneration fails or returns schema-invalid → render the **plain breakdown rail** (v0: raw `breakdown.steps`, unannotated).
2. Breakdown itself is null (trivial single-action task) → render the single `first_tiny_step` as a one-node rail.
3. Whole parse failed → existing FR-002/FR-005 recoverable-error path; raw capture preserved.

Degradation is never silent: the surface shows it fell back to the rail. (Repeated AI failures should surface as Health incidents once NS-INV-6 is wired — do not claim it here.)

### 3.4 Instrumentation writes (NS-INV-3, #235 vocabulary) — node-level, gate-computable

On **map draft presented** (one row):

```
suggestion_records:
  suggestion_type = "task_map_draft"
  subject_type    = "task"
  subject_id      = <task id>
  suggestion_json = { policy_id: "task_map.v1", node_count, speculative_count,
                      nodes: [{order, title, is_speculative, is_optional, on_critical_path}] }
  confidence      = <overall_confidence>
  status          = "pending" -> "accepted" | "rejected" | "ignored" at approve time
```

On **one-pass approve**, resolve the suggestion row (`status`, `resolved_at`) AND write one `override_records` row **per node the user edited, removed, or added** at approve time (this is what makes the §4.4 per-node gate computable):

```
override_records (one per changed node):
  subject_type   = "task_map_node"
  subject_id     = <task id>            -- node identified by order inside new_value_json
  override_type  = "node_removed" | "node_edited" | "node_added"
  old_value_json = <proposed node or null>
  new_value_json = <accepted node or null>   -- includes { order }
  reason         = <optional user note>
```

A node approved **unchanged** produces no override row. Acceptance counts only _proposed_ nodes: `nodes_accepted_unchanged = nodes_proposed − count(overrides WHERE override_type IN (node_removed, node_edited))`. **`node_added` rows are excluded from the acceptance denominator/numerator** — an added node was never proposed, so it must never lower the rate; it is tracked separately as an under-drafting signal (§6, map-missed-work risk). Writes are fire-and-forget (user flow survives learning-write failure).

## 4. v2 Contract Sketch (rolling-wave — NOT full detail)

v2 is a **Stage 2 candidate genuinely dependent on v1 evidence** (issue #292 stage card). Preserve this capability-specific gate: do not implement v2 from this section until v1 evidence satisfies §4.4 and a current contract adopts it.

### 4.1 `task_edges` (additive DAG, sketch)

Normalized cross-task/cross-node dependency graph replacing the v1 denormalized depth-1 jsonb. Sketch shape:

```
task_edges
  id, user_id (owner RLS), area_id
  from_task_id / from_node_order    -- source
  to_task_id   / to_node_order      -- target (enables cross-task edges, depth > 1)
  edge_type   (blocks | informs | speculative)
  created_at
```

Owner RLS + export coverage required in the creating PR (INV-2). Multi-write map transitions use one `SECURITY INVOKER` RPC (INV-1).

### 4.2 Multi-task extraction from one capture

Parse allowed to emit **multiple task drafts wired by `task_edges`** from a single capture (v1 is one-task-per-map). Strict-schema, triage-approved, NS-INV-4.

### 4.3 Map-maintenance loop — trigger points

AI proposes map diffs from execution evidence at: **(a) node completion** (was the next node still right?), **(b) block close / execution session end** (actual vs. estimate drift), **(c) blocker open/close**. Each proposal is a fresh L1 approval (NS-INV-4). This is the "draft-until-approved" revision engine deferred out of v1.

### 4.4 Measurable usage gate (frozen formula)

v2 behavior may not start until, over the gate window (proposed: any rolling 3-week window post-v1):

```
breakdown_node_acceptance =
    Σ nodes_accepted_unchanged  /  Σ nodes_proposed   >= 0.70

where nodes_accepted_unchanged = nodes_proposed
      − count(override_records WHERE subject_type = "task_map_node"
                                 AND override_type IN (node_removed, node_edited))
```

computed by SQL over `suggestion_records` (`policy_id = "task_map.v1"`, `node_count`) and the matching `override_records`. **`node_added` overrides are excluded** — they are not proposed nodes and adding one must never depress acceptance (they feed the separate under-drafting signal). This is why §3.4 records node-level resolution. If the window never qualifies, that is a product signal that the map is over/under-drafting — review v1 usefulness, not skip this evidence-dependent gate.

## 5. Sequencing

**Historical proposed chain: S2 → S3 (#255) → FR-031 v1.** The original draft called this feature FR-022; ADR 0004/current REQUIREMENTS resolved that collision, and the feature was adopted and shipped as FR-031. The original sequencing analysis was:

- **v1 must NOT ride S3's schema bump.** #255's scope is frozen ("MAY touch … extend, not reshape — NS-INV-2; MUST NOT touch … the context-assembly module's interface"). Bundling map fields into S3 widens its frozen scope and its `schema_version` literal semantics. S3 bumps `parse_capture` for `person_mentions` (call it `parse_capture.v3` / `schema_version "1.1"` per S3's own choice); **v1 takes the _next_ additive increment after S3 merges** (`schema_version "1.2"` / prompt `parse_capture.v3+`). Same additive mechanism, separate slice, separate version.
- **v1 depends on S2** because refrain annotations require `contextAssembly.ts` to exist (NS-INV-1). v1 also inherits the S2 charter/operator-profile context via that module's extension point — it wires none of its own.
- **v0** (progression-rail render of the raw breakdown) is prototype-lane / UX-only, no new surface, already in flight; it is v1's degradation target and can proceed independently.

**Completed historical adoption gates:**

- FR-031 and the §2 DATA_MODEL amendment were adopted before implementation; the additive task-map migration and v1 implementation are now shipped.
- The original contract required a dated decision-log entry for extending the frozen S3 parse-result shape and adding task columns; current git history and live artifacts, not this plan, are the evidence of completion.
- The v2 gate threshold (0.70) remains a future capability-specific evidence gate and is not satisfied merely because v1 shipped.

**Goes through current implementation authority first:** adopt and renumber the applicable FR text (§1) and DATA_MODEL amendments (§2) through REQUIREMENTS/ADR review before any implementation slice.

## 6. Risks + recommended defaults

| Risk                                                                             | How the design detects/bounds it                                                                                                                                                                             | Recommended default                                                                                                                                                                 |
| -------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Hallucinated subtasks** (AI invents nodes that don't belong)                   | `is_speculative` flag + node-level `override_records` (`node_removed`) feed the §4.4 acceptance metric; a low acceptance rate _is_ the hallucination alarm.                                                  | Mark low-confidence nodes speculative by default; render them visually distinct; require they be user-kept, not opt-out. Cap at 7 nodes hard.                                       |
| **Under-drafting** (map misses real work the user must add)                      | `node_added` `override_records` are counted separately (excluded from the acceptance rate so they don't mask hallucination); a rising `node_added` rate is the inverse-alarm to hallucinated nodes.          | Track `node_added` per task; a sustained high rate signals the breakdown prompt is too sparse — tune the decomposition, don't raise caps past 7.                                    |
| **Prompt cost** (extra LLM spend)                                                | Map annotation rides the **existing** parse call — no second decomposition (NFR-001). Only explicit regeneration spends a call.                                                                              | On-demand generation only; never background. Cost-optimized model tier; `store: false`.                                                                                             |
| **Map staleness** (plan drifts from reality mid-execution)                       | v1 is **static-after-approve** by design; execution-evidence auto-revision is deferred to v2 behind the usage gate — no stale auto-updates to trust.                                                         | v1: manual user-requested regen only; show `map_approved_at` so age is visible. v2 map-maintenance loop is the real fix, gated on evidence.                                         |
| **Overplanning regression** (owner lingers in the map — the exact governor risk) | Instrument **dwell** (`map_approved_at` → first node `completed_at`/execution-start) and **revision/edit count per task**; rising dwell or edit-churn is the governor's own alarm, visible in decision data. | Collapsed-to-next-node default + one-pass approve are the structural mitigations. If measured dwell trends up, that is a signal to tighten the approve UX, not to add map features. |
| **L2 misread** (reviewer flags one-tap approve as premature autonomy)            | FR-031 explicitly reconciles one-pass approve as L1 (§0.2 / §1 MUST).                                                                                                                                        | Keep every revision a fresh explicit approval; never pre-fill-and-auto-execute.                                                                                                     |
| **Scope creep into v2 during v1 build**                                          | v2 items enumerated as FR-031 Non-goals; `task_edges`, multi-task, maintenance-loop all named as out-of-scope.                                                                                               | Reject any v1 PR that adds cross-task edges, depth > 1, or evidence-triggered revision.                                                                                             |

## Executive summary (10 lines)

1. v1 (**FR-031**) is a thin annotation + persistence layer over the _existing_ `ParseCaptureBreakdown`, not a new node model — the map is a breakdown superset; the v0 rail is its failure-fallback.
2. It adds only three optional per-step fields (`is_speculative`, `is_optional`, `refrain_note`); `on_critical_path` and `kickstart_step`/`first_tiny_step` are reused as the critical flag and two-minute move.
3. Hard caps regardless of trust rung: ~7 nodes, depth 1, on-demand generation, collapsed-to-next-node default.
4. One-pass approve is explicitly **L1** (single suggestion instance), reconciled in the FR so it isn't misread as an L2 autonomy violation of ADR 0002 D1.
5. Refrain annotations flow only through `contextAssembly.ts` (NS-INV-1); the slice wires no context plumbing of its own.
6. Data model: one additive `progression_map jsonb` (+ status/version/approved_at) column on `tasks` — no new table, export coverage inherited (NS-INV-2, INV-2).
7. Born instrumented (NS-INV-3): stable policy id `task_map.v1`, node-level `suggestion_records`/`override_records` so the v2 gate is computable.
8. v2 gate frozen: `Σ nodes_accepted_unchanged / Σ nodes_proposed >= 0.70` — this per-node metric is _why_ v1 logs node-level resolution.
9. v2 (rolling-wave, usage-gated): normalized `task_edges` DAG, multi-task-per-capture, and the execution-evidence map-maintenance loop (node completion / block close / blocker) — all v1 Non-goals.
10. Historical sequencing: **S2 → S3 (#255) → v1**; v1 landed as its own additive schema increment after S3, following REQUIREMENTS-first adoption and the governing review process.
