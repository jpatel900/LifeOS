LifeOS — UI build spec

A precise brief for implementing the redesigned cockpit. The reference prototype is LifeOS Prototype.dc.html. Build it token-driven: dark is the default theme, light is a toggle, and a per-area accent recolors the whole surface from a single variable. One screen component, eight views.

dark-default + light toggle per-area accent token responsive web / mobile add + customize areas
01

Architecture

Theming model

Every color is a semantic CSS custom property set on the app root. Theme switch swaps the base ramp; area switch swaps the accent. Nothing downstream hardcodes a hex. Components read tokens only.

Single component, view router

One container renders the active view via a stage enum. The pipeline spine is both nav and a live workload gauge. No per-page layout duplication — that was the core defect being fixed.

Accent is derived, not stored

Each area stores one base hex. All accent variants (hover, tint, ring, on-accent text) are computed at render so any custom color stays legible in both themes. Formulas in §05.

Glanceability is the bar

State must read pre-cognitively: size-by-load bars, color-by-area cards, count-bearing nodes, progress rings. Avoid reintroducing text-first cards, duplicate state, or multiple equal CTAs.

03

Data model

// one base hex per area; every accent variant is derived Area { name: string, color: hex } // area is an index into areas[]; est in minutes; when is hour 8–18 Item { id, text, area, status: 'inbox'|'today'|'planned'|'done', est?, when? } Session { id, text, outcome: 'complete'|'stuck'|'missed', planned, actual } App { stage: 'today'|'capture'|'triage'|'plan'|'execute'|'review'|'health'|'overview', dark: bool, view: 'web'|'mobile', areas: Area[], areaIndex: int, items: Item[], sessions: Session[], selecting: id|null, // task awaiting a time slot in Plan activeId, running, remaining, total, ending, // focus session (seconds) }

Status lifecycle

inbox → triage decides → today (do), later (someday), or removed (drop). today → assign when → planned. planned → run session → done on complete; stuck/missed log a session but leave status.

Scoping

All six pipeline screens show only the active area's items (item.area === areaIndex). New captures inherit areaIndex. The All areas view ignores the filter and reads everything; the spine counts stay area-scoped.

04

Screens & interactions

Persistent chrome

Header: logo · All areas chip + area chips (each with its color dot; active uses --acc-sf/--acc) + ＋ add-area · accent swatch · Web/Mobile segmented · theme toggle. Spine: 6 nodes (Today, Capture, Triage, Plan, Execute, Review) on a hairline rail. Node circle shows the stage's live count and tone; active node is inverted with an --acc-sf halo. Tapping navigates.

Today

stage: today

Two elements only. One move now — a single card whose content depends on the heaviest unmet stage (inbox→Triage, else today→Plan, else planned→Focus first block, else caught-up). At a glance — a 4-band bar (to triage / to plan / scheduled / done); each band's width = max(1,count)+0.6, colored by stage tone. Bands and card CTA navigate.

Capture

stage: capture

Centered borderless textarea (caret = --acc), one Save thought button, ⌘↵ shortcut. On save: push inbox item in active area, clear field, flash a green "waiting in Triage" toast (~1.4s). Area picker / save-mode / drafts deliberately hidden behind /.

Triage

stage: triage

One card at a time (first inbox item) + a dot row showing how many remain. Three directional targets: Drop (remove), Someday (→later), Do today (→today, accent-filled). Next card slides up. Empty → "Inbox clear" with a Plan CTA.

Plan

stage: plan

Two-pane: hour rail (8am–6pm) + "To place" list. Tap a task → selecting set, empty hours show "drop here" with an --acc tint; tap an hour → item becomes planned with that when. Tap a placed block to unplan. All placed → "Start focusing" CTA. Mobile: panes stack.

Execute

stage: execute

Picker lists planned blocks (time chip + est). Choosing one starts a session: total = est×60. Countdown ring (r=104) depletes; one Start/Pause/Resume + End session. At zero or End → outcome step (Complete→done, Stuck, Missed); all log a Session with planned vs actual minutes.

Review

stage: review

Verdict-first: completion ring (done / total) + headline ("day closed clean" vs "N carry over"). Below: planned-vs-actual focus bars (from sessions) and outcome pills. Lead with the answer; counts are support, not the headline.

Health

stage: health

Single green status ring + "All systems healthy · 11/11". Three grouped summary tiles (Storage, Integrations, Telemetry-off) instead of 11 cards. "Run system check" pulses the ring ~1.4s. Full breakdown is one tap away, not shown by default.

All areas

stage: overview

Cross-area mission control. Workload spectrum — one bar split by area color, each segment = that area's open count / total. Legend with per-area open counts. Pipeline board — 4 columns (To triage / To plan / Scheduled / Done) holding every item from every area; each card is tinted mix(areaColor,--sf2) with a 3px area-color left edge and a labeled area dot. Mobile: columns stack.

05

Derived-value formulas

mix(a,b,t) // per-channel linear blend of two hex colors lum(hex) // WCAG relative luminance (sRGB→linear, 0.2126/0.7152/0.0722) acc2 = mix(acc, dark ? '#fff' : '#000', 0.16) acc-sf = mix(acc, sf2, dark ? 0.80 : 0.86) acc-rng = mix(acc, sf2, dark ? 0.50 : 0.66) on-acc = lum(acc) > 0.55 ? '#1a1a14' : '#ffffff' cardBg = mix(areaColor, sf2, dark ? 0.83 : 0.88) // overview cards ring r=104 (focus), r=86 (review) dash = 2·π·r offset = dash · (1 − value/total) // value=remaining or done band.flex = max(1, count) + 0.6 // Today glance + spectrum weighting

Notes for the agent: keep accent derivation in one util so a custom area color can never produce illegible text — on-acc guards contrast on solid fills. Persist areas, items, sessions, dark, and areaIndex. Session timer is the only wall-clock dependency. The prototype keeps state in memory; swap for your store without touching the view layer.
